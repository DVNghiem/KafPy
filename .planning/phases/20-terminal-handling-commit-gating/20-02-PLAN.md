Add `flush_failed_to_dlq` method to the `OffsetCoordinator` trait and implement it in `OffsetTracker`.

The method signature:
```rust
fn flush_failed_to_dlq(
    &self,
    dlq_router: &Arc<dyn DlqRouter>,
    dlq_producer: &Arc<SharedDlqProducer>,
);
```

Implementation:
1. Add method declaration to `OffsetCoordinator` trait in `src/coordinator/offset_coordinator.rs`
2. Implement in `OffsetTracker`:
   - Iterate all partitions via `all_partitions()`
   - For each partition, get `failed_offsets` BTreeSet
   - For each failed offset, produce to DLQ using `dlq_router.route()` and `dlq_producer.produce_async()`
   - Construct `DlqMetadata` with original topic/partition/offset, reason from `last_failure_reason`, attempt_count=1, timestamps
   - Log each flush action
3. **Key design (D-02)**: This method flushes ALL failed (retryable + terminal) — failed_offsets includes both types
4. **Note**: OffsetTracker does not have access to the original message payload/key — use empty payload for now since SharedDlqProducer needs it. The actual DLQ content is a known limitation — logged as warning.

This is a fire-and-forget flush — don't block waiting for broker acknowledgment.

Current shutdown signature:
```rust
pub async fn shutdown(&mut self) {
    tracing::info!("initiating worker pool shutdown");
    self.shutdown_token.cancel();

```

New shutdown:
```rust
pub async fn shutdown(&mut self) {
    tracing::info!("initiating worker pool shutdown");
    self.shutdown_token.cancel();
    // D-02: Flush all failed (retryable + terminal) to DLQ before final commit
    self.offset_coordinator.flush_failed_to_dlq(&self.dlq_router, &self.dlq_producer);
    self.offset_coordinator.graceful_shutdown();
    self.join_set.shutdown().await;
    tracing::info!("worker pool shutdown complete");
}
```

Also update the WorkerPool struct to store `dlq_router` and `dlq_producer` with `pub(crate)` visibility so they can be accessed in `shutdown()` — currently they are stored but not accessible for the flush call (struct has private fields, but we call methods on the Arc refs directly).
