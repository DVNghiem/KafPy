# KafPy Benchmark Methodology

## Overview
Brief description of the benchmark framework's purpose and how run_scenario() works.

## Percentile Computation (NOTE-01)

### P50 / P95 / P99 Computation
Explain how percentiles are computed:
- KafPy uses t-digest algorithm (via HistogramRecorder) for accurate high-percentile estimation
- t-digest is a tree-based histogram that maintains accuracy at the tails (unlike naive sorted arrays)
- Percentile computation: `histogram.percentile(50.0)` returns the value at the 50th percentile
- The t-digest compression factor controls the memory-accuracy tradeoff; KafPy uses the default compression

### Why t-digest?
- Naive approaches (sorted array) require O(n log n) per flush and O(n) memory
- t-digest uses O(k) memory where k is the compression factor
- Accuracy: within 0.5% of true percentile at p99 with default compression

### What the Percentiles Mean
- P50 (median): 50% of messages are processed faster than this
- P95: 95% of messages are processed faster than this; used for SLA targets
- P99: 99% of messages are processed faster than this; used for tail-latency detection
- P99.9: available for extreme tail analysis (configurable via PercentileBuckets)

## Warmup Exclusion (NOTE-01)

### How Warmup Works
Explain the warmup exclusion strategy:
- First N messages (warmup_messages, default 1000) are excluded from latency/throughput statistics
- This eliminates JIT compilation effects (Python/Rust warmup), connection establishment overhead, and Kafka producer consumer group rebalancing from measured metrics
- Warmup is configured per scenario: `warmup_messages` parameter in ScenarioConfig

### Configuring Warmup
- Default: 1000 messages for most scenarios
- Long-running scenarios: warmup_messages can be increased (e.g., 5000) to ensure JIT is fully optimized
- Short scenarios: ensure warmup_messages < 10% of total messages

Example:
```python
result = kafpy.benchmark.run_scenario(
    "throughput",
    num_messages=50_000,
    warmup_messages=5000,  # Exclude first 5000 from measurement
)
```

## Confidence Intervals (NOTE-01)

### T-Digest Accuracy
Explain the statistical properties:
- t-digest provides relative accuracy guarantees: within d% of the true percentile where d is determined by the compression factor
- At compression=100 (default): p99 estimates are within ~1% of true value for 10M samples
- Confidence widens for extreme percentiles (p99.9) but remains bounded

### Run-to-Run Variation
- Latency percentiles vary run-to-run due to OS scheduler, network jitter, Kafka broker load
- Recommendation: run each scenario at least 3 times and report the median
- For CI regression detection: compare against baseline using comparison reports (see TUNING.md)

## Reproducibility Requirements (NOTE-01, NOTE-04)

### Kafka Broker Requirements
Document the assumed Kafka broker setup (NOTE-04):
- Broker version: Kafka 3.x or later recommended
- The benchmark assumes the broker under test is dedicated or has controlled noise
- Broker metrics (CPU, disk, network) can affect results significantly
- For accurate latency benchmarks: use a broker on the same LAN as the benchmark client (< 1ms network latency)

### Network Latency Contribution (NOTE-04)
Explain how network latency affects results:
- Network latency adds directly to round-trip time: measured_latency = processing_latency + network_latency
- For localhost benchmarks: ~0.1-0.5ms per hop
- For remote broker: measure network latency separately with `ping` and subtract from reported p50
- Rule of thumb: if broker network latency > 5ms, latency benchmarks will be dominated by network, not KafPy

### Payload Size Impact (NOTE-04)
Explain how message size affects benchmarks:
- Larger payloads increase serialization/deserialization cost (proportional to size)
- Throughput benchmarks: smaller payloads are typically the bottleneck (KafPy overhead per message is constant)
- Latency benchmarks: payload size has minimal impact on p50 (fixed overhead dominates) but can affect p99 (GC pressure from large object handling)
- Recommendation: test with payload sizes matching your production workload (256B to 1MB range)

### Reproducibility Checklist
- Use the same Kafka broker (same version, same configuration) across runs
- Use the same network path (localhost for lowest variance)
- Lock or disable power management on the benchmark machine
- Use the same num_messages and warmup_messages settings
- Run multiple iterations and use the median result
- Record broker metrics (CPU, disk IO) alongside benchmark results

## Benchmark Types

### Throughput Scenario
Describe throughput scenario behavior:
- Measures maximum message throughput (msg/s)
- Rate=None means unlimited (producer goes as fast as possible)
- Metrics: throughput_msg_s is the primary metric

### Latency Scenario
Describe latency scenario behavior:
- Measures round-trip latency under steady-state load
- Background load is generated at the configured `rate`
- Metrics: latency_p50_ms, latency_p95_ms, latency_p99_ms are the primary metrics

### Failure Scenario
Describe failure scenario behavior:
- Injects failures at the configured `failure_rate` (0.0 to 1.0)
- Measures DLQ routing correctness and retry budget enforcement
- Metrics: error_rate should match failure_rate if DLQ routing works correctly

# KafPy Benchmark Tuning Guide

## Overview
Practical guidance for tuning KafPy consumer parameters to achieve optimal benchmark results and production performance.

## Key Tuning Parameters

### queue_depth (NOTE-02)
Explain queue_depth tuning:
- Controls the maximum number of messages buffered in the consumer channel
- Default: 1000
- Impact: too low causes backpressure (consumer waits for handler), too high causes memory bloat
- For high-throughput benchmarks: increase to 5000-10000 if broker can sustain the rate
- For latency benchmarks: keep at 100-500 to minimize queueing delay
- Monitoring: watch `kafpy_handler_queue_size` metric

```python
config = kafpy.ConsumerConfig(
    bootstrap_servers="localhost:9092",
    group_id="benchmark",
    topics=["benchmark"],
    # Tunable: increase for throughput, decrease for latency
)
```

### concurrent_handlers (NOTE-02)
Explain concurrent_handlers tuning:
- Number of Tokio tasks that poll the handler queues and invoke Python callbacks
- Default: num_cpus::get() (number of CPU cores)
- Impact: too few leaves CPU cores idle, too many causes context-switch overhead
- For CPU-bound handlers (minimal I/O): set to num_cpus
- For I/O-bound handlers (network calls): can exceed num_cpus (e.g., 2x num_cpus)
- Benchmark note: concurrent_handlers affects throughput but not latency percentiles

### batch_size (NOTE-02)
Explain batch_size tuning:
- Number of messages to batch before invoking the Python handler
- Default: 1 (single-message processing)
- Impact: larger batches increase throughput by amortizing Python callback overhead, but increase latency
- For throughput benchmarks: try batch_size=50, 100, 500
- For latency benchmarks: keep batch_size=1
- With batch_size > 1: latency is measured per-batch, not per-message

```python
batch_config = kafpy.BatchConfig(
    max_size=100,      # Tune for throughput vs latency tradeoff
    max_latency_ms=10, # Max wait time before flushing partial batch
)
```

### timeout Settings (NOTE-02)
Explain timeout tuning:
- `fetch_timeout`: how long the Kafka consumer waits for a fetch batch
- `handler_timeout`: how long a handler can run before being cancelled
- `shutdown_timeout`: how long graceful shutdown waits before force-exit
- Default fetch_timeout: 500ms (sweet spot for latency vs throughput)
- For latency benchmarks: reduce to 100ms for faster fetch cycles
- For throughput benchmarks: increase to 1000ms to allow larger fetch batches
- handler_timeout: should exceed your 99th-percentile handler duration

## Scenario-Specific Recommendations

### Throughput Benchmark Tuning
Recommended settings for maximum throughput:
- queue_depth: 10000
- concurrent_handlers: num_cpus * 2
- batch_size: 100
- warmup_messages: 10000
- num_messages: 1_000_000 (larger sample for accurate tail percentiles)

### Latency Benchmark Tuning
Recommended settings for accurate latency measurement:
- queue_depth: 100
- concurrent_handlers: 1
- batch_size: 1
- rate: 10000 (steady background load)
- warmup_messages: 1000
- num_messages: 100_000

### Failure/Benchmark Tuning
Settings for measuring DLQ and retry behavior:
- failure_rate: 0.01 to 0.05 (1-5% failure injection)
- Verify error_rate in result matches failure_rate
- Check DLQ topic for correctly routed messages

## Broker-Side Tuning
Explain broker-side requirements for accurate benchmarks:
- `num.network.threads`: match or exceed client concurrency
- `queued.max.requests`: should exceed queue_depth (default 500)
- `socket.send.buffer.bytes`: increase for high-throughput benchmarks (e.g., 1MB)
- `socket.recv.buffer.bytes`: increase for high-throughput benchmarks (e.g., 1MB)
- `log.segment.bytes`: larger segments reduce metadata overhead

## Cross-Reference to Methodology
Refer readers to BENCHMARK-METHODOLOGY.md for:
- How warmup exclusion works
- How t-digest affects percentile accuracy
- How to interpret confidence intervals
- Reproducibility requirements (broker version, network path, etc.)
