**StreamingState enum:**
```rust
/// State machine for streaming handler lifecycle.
/// Transitions: Starting → Running → Draining → Recovering
#[derive(Debug, Clone, PartialEq, Eq)]
enum StreamingState {
    /// Subscribing to topic, creating async generator
    Starting,
    /// Polling messages and yielding to consumer
    Running,
    /// Cancellation received, draining buffered messages
    Draining,
    /// Error detected, applying retry/backoff before Restart
    Recovering { attempt: u32 },
}
```

**streaming_worker_loop function signature:**
```rust
pub async fn streaming_worker_loop(
    worker_id: usize,
    consumer: StreamConsumer,
    handler: PythonHandler,
    cancel: CancellationToken,
    queue_manager: Arc<QueueManager>,
) -> Result<(), WorkerError>
```

**Implementation structure (4 phases):**

1. **STARTING phase:**
   - Log worker starting
   - Create ExecutionContext
   - Subscribe to Kafka topic
   - Transition to Running

2. **RUNNING phase (main loop):**
   ```rust
   loop {
       // Check cancellation first (D-04: coordinated with graceful shutdown)
       if cancel.is_cancelled() {
           state = StreamingState::Draining;
           break;
       }

       // Poll the async generator via invoke_streaming
       match handler.invoke_streaming(&ctx, message).await {
           ExecutionResult::Ok => {
               // Generator exhausted normally — graceful stop
               break;
           }
           ExecutionResult::Error { reason, exception, traceback } => {
               // Error during yield — error recovery path (D-03)
               state = StreamingState::Recovering { attempt: 1 };
               // Apply backoff, retry up to max_attempts
               // If max retries exceeded, break and exit
           }
           _ => {
               // Unexpected result — treat as error
               break;
           }
       }
   }
   ```

3. **DRAINING phase:**
   - Call `coro.close()` on the generator (via PythonAsyncFuture Drop)
   - Commit any buffered offsets
   - Log drain completion
   - Exit loop

4. **RECOVERING phase:**
   - Apply exponential backoff: `tokio::time::sleep(Duration::from_secs(2_u64.pow(attempt)))`
   - Increment attempt counter
   - If attempt < max_retries: transition to Running
   - If attempt >= max_retries: transition to Draining then exit

**Error handling:**
- Import `WorkerError` from mod.rs
- Import `tokio_util::sync::CancellationToken`
- Import `crate::python::execution_result::ExecutionResult`
- Import `crate::python::handler::PythonHandler`
- Import `rdkafka::consumer::StreamConsumer`

**Key insight:** The main loop runs inside the worker task (spawned by WorkerPool). CancellationToken::is_cancelled() is checked at the top of each iteration. When cancelled, the PythonAsyncFuture is dropped which calls coro.close() via its Drop impl (async_bridge.rs:92-99).
