  <name>Task 5: Implement W3C tracecontext propagation across spawn_blocking boundary</name>
  <files>src/python/handler.rs</files>
  <action>
In src/python/handler.rs, modify the spawn_blocking call sites to propagate W3C tracecontext:

1. **Import at top of file**:
   ```rust
   use crate::observability::tracing::{inject_trace_context, extract_trace_headers};
   use std::collections::HashMap;
   ```

2. **In invoke() method** (line 161-219), around the spawn_blocking call:
   ```rust
   // BEFORE entering spawn_blocking: extract current trace context
   let trace_headers = extract_trace_headers();

   let result = tokio::task::spawn_blocking(move || {
       // BEFORE calling Python: inject trace context into Python dict
       // (The Python dict is built before the spawn_blocking, but we inject headers there)

       Python::attach(|py| {
           // ... existing code to build py_msg dict ...

           // Add trace context headers to the Python dict
           let headers_dict = PyDict::new(py);
           for (key, value) in trace_headers {
               let _ = headers_dict.set_item(&amp;key, &amp;value);
           }
           let _ = py_msg.set_item("trace_context", headers_dict);

           // ... rest of callback call ...
       })
   }).await;
   ```

   Wait — this approach adds trace_context to the Python message dict, but the callback is invoked AFTER headers are set. Let me reconsider.

   Actually, the W3C tracecontext needs to propagate INTO the Python handler and then BACK. The pattern is:

   - Rust side (before spawn_blocking): Extract current span context as headers
   - spawn_blocking closure: Inject headers into a HashMap, pass to Python
   - Python callable: Uses those headers to create a child span (Python instrumentation is deferred to v1.8 — OBS-F1)
   - Return: Extract any outgoing headers from Python result

   Since Python-side tracing is deferred, the immediate need is:
   - Inject trace context into headers that Python can read
   - The Python handler uses those headers if it wants (user's responsibility in v1.7)

   The simplest approach for v1.7:
   ```rust
   let trace_headers = extract_trace_headers();

   let result = tokio::task::spawn_blocking(move || {
       Python::attach(|py| {
           // Add trace context as headers in the message dict
           let trace_dict = PyDict::new(py);
           for (key, value) in &amp;trace_headers {
               let _ = trace_dict.set_item(key, value);
           }
           let _ = py_msg.set_item("trace_context", trace_dict);

           // ... rest of invoke logic ...
       })
   }).await;
   ```

   Actually, the headers should be added to the message BEFORE the callback is called. But py_msg is built before spawn_blocking. Let me look at the actual code flow.

   Looking at lines 161-219:
   - py_msg dict is built INSIDE the spawn_blocking closure
   - So trace_headers must be captured from outside the closure

   Correct approach:
   ```rust
   let trace_headers = extract_trace_headers();

   let result = tokio::task::spawn_blocking(move || {
       Python::attach(|py| {
           let py_msg = PyDict::new(py);
           // ... existing fields ...

           // Add trace context headers as a nested dict
           let trace_dict = PyDict::new(py);
           for (key, value) in &amp;trace_headers {
               let _ = trace_dict.set_item(key, value);
           }
           let _ = py_msg.set_item("_trace_context", trace_dict);

           // ... rest of callback call ...
       })
   }).await;
   ```

3. **In invoke_batch() method** (line 226-283), same pattern:
   ```rust
   let trace_headers = extract_trace_headers();

   let result = tokio::task::spawn_blocking(move || {
       Python::attach(|py| {
           // ... existing batch building ...

           // Add trace context to each message dict in the batch
           let trace_dict = PyDict::new(py);
           for (key, value) in &amp;trace_headers {
               let _ = trace_dict.set_item(key, value);
           }
           let _ = py_msg.set_item("_trace_context", trace_dict);

           // ... rest of batch invoke ...
       })
   }).await;
   ```

4. **In invoke_async() and invoke_batch_async()** — these use PythonAsyncFuture which is NOT spawn_blocking. The trace context propagates through the Future poll via the standard tracing context. No extra work needed for W3C propagation here since we're not crossing the GIL boundary in the same way.

OBS-16: Span context propagates across PyO3 GIL boundary via explicit W3C tracecontext header injection at spawn_blocking call sites.

Note: The actual Python-side propagation (using these headers to continue the trace) is deferred to v1.8 per OBS-F1. In v1.7, we just make the headers available to Python.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| worker_loop -> invoke_mode | handler_id, topic, partition, offset, mode are metadata not payload; span fields contain no PII |
| spawn_blocking GIL boundary | W3C tracecontext headers propagated; Python handler receives trace context (user's responsibility in v1.7) |
| ConsumerDispatcher dispatch | topic, partition, offset from Kafka message metadata; routing_decision derived from dispatch outcome |
| DLQ routing | handler_id, reason, partition from metadata and failure reason; no user payload in span |

| Threat ID | Category | Component | Disposition | Mitigation Plan |
