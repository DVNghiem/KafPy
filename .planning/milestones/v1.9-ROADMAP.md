## Phase Details

### Phase 38: Result Models & Measurement Infrastructure
**Goal**: Data models for benchmark results and measurement helpers for collecting latency/throughput/memory samples off the hot path.
**Depends on**: Nothing (first phase)
**Requirements**: RES-01, RES-02, RES-03, RES-04, RES-05, RES-06, MEAS-01, MEAS-02, MEAS-03, MEAS-04, MEAS-05, MEAS-06, MEAS-07, MEAS-08
**Success Criteria** (what must be TRUE):
  1. BenchmarkResult struct captures total_messages, duration_ms, throughput_msg_s, latency_p50/p95/p99_ms, error_rate, memory_delta_bytes
  2. AggregatedResult holds mean/stddev/min/max across multiple runs
  3. MeasurementStats computes mean/variance/percentiles online (counter, sum, sum_squared, min, max, count)
  4. HistogramRecorder uses t-digest algorithm for accurate high-percentile computation
  5. LatencyTimer uses Instant for nanosecond-precision scoped timing
  6. ThroughputMeter tracks messages/bytes and computes rate on demand
  7. MemorySnapshot uses RuntimeSnapshot for heap_allocated delta between snapshots
  8. All measurement code aggregates off hot path via background task; warmup exclusion configurable
  9. MetricLabels from observability subsystem reused for benchmark metric labels
  10. All result types implement Serialize/Deserialize via serde; BenchmarkResult and AggregatedResult implement CsvSerializable

### Phase 39: Scenario Definitions
**Goal**: Scenario trait and concrete workload profiles defining what to benchmark, configurable via Python dict or TOML.
**Depends on**: Phase 38
**Requirements**: SCEN-01, SCEN-02, SCEN-03, SCEN-04, SCEN-05, SCEN-06, SCEN-07, SCEN-08
**Success Criteria** (what must be TRUE):
  1. Scenario trait defines WHAT to benchmark: target topic, message rate, payload size, duration, warmup messages
  2. WorkloadProfile enum has variants: ThroughputFocused, LatencyFocused, FailureFocused, BatchComparison, HandlerModeComparison
  3. ThroughputScenario, LatencyScenario, FailureScenario, BatchVsSyncScenario, AsyncVsSyncScenario all implement Scenario
  4. All scenarios are configurable via Python dict or TOML file without modifying runner core
  5. New scenarios can be added by implementing Scenario trait without touching runner

### Phase 40: Benchmark Runner
**Goal**: Orchestrates scenario execution, warmup, measurement window, teardown with graceful termination.
**Depends on**: Phase 38, Phase 39
**Requirements**: RUN-01, RUN-02, RUN-03, RUN-04, RUN-05, RUN-06, RUN-07
**Success Criteria** (what must be TRUE):
  1. BenchmarkRunner.run_scenario(scenario) -> BenchmarkResult is the primary entry point
  2. BenchmarkContext provides message generator, config access, result writer to scenarios
  3. Runner accepts MetricsSink for metric emission; benchmark has its own sink for self-measurement
  4. Runner drains inflight messages and commits offsets before shutting down (graceful termination)
  5. BenchmarkRunner is Send + Sync for use from Python async context via PyO3
  6. `python -m kafpy.benchmark run --scenario throughput --output ./results` CLI works

### Phase 41: Result Output
**Goal**: Machine-readable and human-readable output formats for benchmark results.
**Depends on**: Phase 38
**Requirements**: OUT-01, OUT-02, OUT-03, OUT-04, OUT-05, OUT-06
**Success Criteria** (what must be TRUE):
  1. JSON output: one file per run named bench-{scenario}-{timestamp}.json
  2. CSV output: bench-{scenario}-{timestamp}.csv with one row per measurement interval
  3. BenchmarkReport is human-readable markdown with scenario name, key metrics, baseline comparison
  4. compare(base, current) -> ComparisonReport shows delta %, highlights regressions
  5. Output path configurable, defaults to ./benchmark_results/
  6. Machine-readable diff: bench-diff-{scenario}-{timestamp}.json with before/after values and delta

### Phase 42: Hardening Checks
**Goal**: Production hardening validations ensuring no message loss, bounded memory, graceful shutdown, correct DLQ routing.
**Depends on**: Phase 38
**Requirements**: HARD-01, HARD-02, HARD-03, HARD-04, HARD-05, HARD-06, HARD-07, HARD-08
**Success Criteria** (what must be TRUE):
  1. HardeningCheck enum variants: BackpressureThreshold, MemoryLeakCheck, GracefulShutdownCheck, DlqDrainCheck, RetryBudgetCheck
  2. ValidationResult struct captures check name, passed bool, details, suggestions vec
  3. HardeningRunner.run_all() -> Vec<ValidationResult> runs all checks
  4. Backpressure check: consumer honors queue_depth limits without message loss under saturated load
  5. Memory leak check: no significant heap growth (> 1MB delta) over sustained 10M message run
  6. Graceful shutdown check: pending messages processed and offsets committed before exit on SIGINT
  7. DLQ drain check: all failed messages delivered to DLQ topic after graceful shutdown
  8. Retry budget check: messages exhaust retry budget and route to DLQ (not infinite retry loop)
**Plans**: TBD

**Success Criteria** (what must be TRUE):
  1. kafpy.benchmark module exposes run_scenario, BenchmarkResult, ScenarioConfig, BenchmarkReport, run_hardening_checks
  2. ScenarioConfig dataclass (Python): scenario_type, num_messages, payload_bytes, rate, warmup_messages, failure_rate
  3. BenchmarkReport dataclass (Python): scenario name, metrics dict, passed checks, suggestions list
  4. All benchmark result types frozen after construction; no mutation after result is written
  5. kafpy.benchmark defines its own __all__ listing only intended public API
  6. BENCHMARK-METHODOLOGY.md at repo root: P50/P95/P99 computation, warmup exclusion, confidence intervals, reproducibility requirements
  7. TUNING.md with practical guidance on queue_depth, concurrent_handlers, batch_size, timeout settings
  8. Methodology doc covers assumptions: Kafka broker under test, network latency contribution, payload size impact
