## PythonAsyncFuture — the custom Future type

```rust
// Stored as Arc<Py<PyAny>> (same as callback on PythonHandler)
// Implements Future with Output = ExecutionResult
pub struct PythonAsyncFuture {
    coro: Py<PyAny>,        // The Python coroutine object
    polled: bool,           // Ensures coro.send(None) is called exactly once
}

impl Future for PythonAsyncFuture {
    type Output = ExecutionResult;
    fn poll(self: Pin<&mut Self>, cx: &mut Context) -> Poll<Self::Output> { ... }
}
```

## Coroutine polling behavior

| State | On poll | Return |
|-------|---------|--------|
| Not yet polled | `coro.send(None)` | `Poll::Pending`, schedule wake |
| Yielded (Pending) | `coro.send(None)` again | `Poll::Pending`, schedule wake |
| Returned normally | `StopIteration` caught | `Poll::Ready(Ok)` |
| Raised exception | `PyErr` caught | `Poll::Ready(Error { reason, exception, traceback })` |
</context>

Create `src/python/async_bridge.rs` with:

1. **`PythonAsyncFuture` struct** — stores `coro: Py<PyAny>` (the coroutine object) and `polled: bool`

2. **`Future` implementation** — `poll` method:
   - If already polled and not ready, the coroutine was already advanced — this state should not occur if used correctly (coro.send(None) called once per poll cycle)
   - Acquire GIL via `Python::with_gil(|py| { ... })`
   - Call `coro.call_method0(py, "send", py.None())` to advance the coroutine
   - On `Ok(val)` — coroutine yielded; register waker with `cx.waker().wake_by_ref()`, return `Poll::Pending`
   - On `Err(py_err)` if `py_err.is_instance_of::<pyo3::types::PyStopAsyncIteration>()` or `StopIteration` — coroutine finished normally; return `Poll::Ready(ExecutionResult::Ok)`
   - On `Err(py_err)` for any other exception — extract exception name and traceback, return `Poll::Ready(ExecutionResult::Error { reason: FailureReason::Terminal(HandlerPanic), exception, traceback })`

3. **Handle StopIteration** — Python coroutines raise StopIteration when they return normally. Catch this explicitly:
   ```rust
   if py_err.is_instance_of::<pyo3::types::PyStopIteration>() {
       return Poll::Ready(ExecutionResult::Ok);
   }
   ```

4. **Waker registration** — Use `Arc::new(Waker)` as the waker for the Python coroutine. Since Python coroutines don't natively integrate with Tokio, the waker does nothing (always wakes on next poll) — this is acceptable because the Tokio runtime will re-poll when the task is woken from Rust-side.

5. **Drop handling** — When `PythonAsyncFuture` is dropped, ensure the coroutine is closed:
   ```rust
   impl Drop for PythonAsyncFuture {
       fn drop(&mut self) {
           Python::with_gil(|py| {
               let coro = self.coro.as_ref(py);
               let _ = coro.call_method0("close");
           });
       }
   }
   ```

6. **`From<Py<PyAny>>` for `PythonAsyncFuture`** — constructor that takes ownership of the coroutine object.

7. **Import `FailureReason`, `HandlerPanic`, `ExecutionResult`** from the existing crate.

**Key implementation notes:**
- Use `Python::with_gil` to acquire GIL only during `coro.send(None)` — GIL is released immediately after
- Do NOT use `spawn_blocking` — the async path must NOT hold GIL across await points
- The `coro` object is `Py<PyAny>` (same as callback storage) — GIL-independent, Send+Sync
- The coroutine is already created by the time `PythonAsyncFuture` is constructed — we just wrap it
