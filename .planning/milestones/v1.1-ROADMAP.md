# Milestone v1.1: Async & Concurrency Hardening

**Status:** ✅ SHIPPED 2026-04-29
**Phases:** 7-10
**Total Plans:** 10

## Overview

Prevent long-running sync handlers from blocking the poll cycle; expand Python handler API with streaming patterns, middleware, timeouts, and async fan-in/out.

## Phases

### Phase 7: Thread Pool

**Goal**: Sync handlers execute on Rayon work-stealing pool instead of blocking Tokio's poll cycle, preventing heartbeat misses and rebalances.
**Depends on**: Phase 6 (Hardening)
**Plans**: 1 plan

Plans:

- [x] 07-01: RayonPool struct, rayon_pool_size config, PythonHandler dispatch

**Details:**
- Rayon work-stealing pool for blocking sync handlers (non-blocking poll)
- ConsumerConfigBuilder::rayon_pool_size(u32) configuration
- Graceful shutdown coordinates with Rayon drain (30s timeout)

### Phase 8: Async Timeout

**Goal**: Async handlers can be aborted after a configured timeout, with timeout metadata in DLQ and timeout metrics in Prometheus.
**Depends on**: Phase 7 (Thread Pool)
**Plans**: 2 plans

Plans:

- [x] 08-01: TimeoutInfo struct, ExecutionResult::Timeout, DlqMetadata timeout fields, TimeoutMetrics counter
- [x] 08-02: handle_execution_failure enriched, TimeoutMetrics emission in worker loop

**Details:**
- @handler(topic, timeout=X) Python API — timeout chain from add_handler to invoke_mode_with_timeout
- Timeout metadata propagated to DLQ envelope (timeout_duration, last_processed_offset)
- Prometheus counter kafpy.handler.timeout_total increments on handler timeout

### Phase 9: Handler Middleware

**Goal**: Handler middleware chain enables before/after/on_error hooks for cross-cutting concerns (logging, metrics) without duplicating handler code.
**Depends on**: Phase 8 (Async Timeout)
**Plans**: 3 plans

Plans:

- [x] 09-01: HandlerMiddleware trait + MiddlewareChain
- [x] 09-02: Logging + Metrics built-in middleware
- [x] 09-03: Python API middleware per handler

**Details:**
- HandlerMiddleware trait with before/after/on_error hooks
- Built-in Logging middleware (span events on handler start/complete/error)
- Built-in Metrics middleware (latency histogram, throughput counter)
- Python API @handler(middleware=[Logging(), Metrics()]) per handler

### Phase 10: Streaming Handler

**Goal**: Persistent async iterable handlers maintain long-lived connections (WebSocket, SSE, live dashboard feeds) with proper lifecycle and backpressure.
**Depends on**: Phase 9 (Handler Middleware)
**Plans**: 4 plans

Plans:

- [x] 10-01: HandlerMode::StreamingAsync + invoke_streaming
- [x] 10-02: @stream_handler Python API
- [x] 10-03: streaming_worker_loop lifecycle
- [x] 10-04: backpressure propagation

**Details:**
- HandlerMode::StreamingAsync variant for persistent async iterable handlers
- @stream_handler(topic) Python API — long-lived handler that loops on async iterator
- Lifecycle management: start/subscribe, run/loop, stop/drain, error recovery with backoff
- Per-stream backpressure: slow consumer pauses Kafka consumption

## Milestone Summary

**Key Decisions:**

- Rayon work-stealing pool for sync handlers (Rationale: Prevent poll cycle blocking, heartbeat/rebalance risk)
- Tokio stays as async runtime (Rationale: No change to existing async infrastructure)
- Python GIL calls via spawn_blocking inside Rayon closures (Rationale: GIL safety preserved)
- Oneshot channels for Tokio-Rayon communication (Rationale: Prevent deadlock — no Tokio APIs from Rayon)

**Issues Resolved:**

- Long-running sync handlers blocking Tokio poll cycle (causing heartbeat misses)
- No structured timeout propagation (now uses ExecutionResult::Timeout)
- No middleware system for cross-cutting concerns
- No streaming handler support for persistent connections

**Issues Deferred:**

- Phase 7 (Thread Pool) verification shows drain()/abort() are no-ops (Rayon limitation, documented)
- Async fan-in/fan-out (FANIN/FANOUT requirements) deferred to v2.0

**Technical Debt Incurred:**

- streaming_worker_loop State::Draining path may need refinement for production edge cases
- PausePartition/ResumePartition backpressure actions untested with actual Kafka consumer
